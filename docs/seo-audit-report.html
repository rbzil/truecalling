Executive summary

TrueCalling has a solid SSR foundation (Next.js 14 App Router on Vercel), correct per-locale title differentiation, and a working 8-locale URL structure with translated slugs. The site is not yet ready for production SEO because of five blocking technical issues that compound across locales: a global canonical collapse points every sub-page at the locale homepage; the staging Vercel environment is openly indexable in parallel with the canonical domain; robots.txt returns 404 because the locale catch-all route intercepts the path; the root <html lang> attribute is hardcoded to English on every locale page; and broken hreflang on sub-routes points at the wrong target URLs.

Beyond technical correctness, the homepage is the wrong page type for the primary commercial keywords ("AI sourcing software", "recruiting AI tool") — Google rewards comparison guides and listicles for these queries (90 % of top 10 results). A separate comparison content asset is required to capture organic traffic for these terms; the homepage will only ever rank for branded and bottom-of-funnel queries.

The blog is the strongest GEO asset on the site, but it is undermined by missing author attribution, unsourced statistics, and absent image on BlogPosting schema (which alone makes every article ineligible for Google rich results). The site has no llms.txt, no Organization schema, and no AI crawler directives — limiting LLM citation potential at a moment where Perplexity and ChatGPT Search are becoming meaningful traffic sources for B2B SaaS.

Critical issues

Critical 1   Canonical collapse on all sub-routes

Every sub-route (/en/book-a-demo, /en/contact, /en/faq, /fr/reserver-une-demo, /fr/contact, /fr/politique-de-confidentialite, etc.) issues a <link rel="canonical"> pointing back at the locale homepage instead of itself. Google will treat each as a duplicate of the home and refuse to index ~48 URLs across the 8 locales independently. Only /en/blog self-canonicalizes correctly because it has its own generateMetadata.

Root cause: the generateMetadata in app/[locale]/layout.tsx sets canonical to the locale home; all sub-routes inherit it because they don't override it.

Fix: add a generateMetadata() export to every sub-page module (book-a-demo, contact, faq, terms, legal-notice, privacy, cookies, blog/[slug]) that returns alternates.canonical = the page's own URL, computed via getLocalizedPath(routeKey, locale).

Critical 2   Hreflang sub-routes point at wrong URLs

On every sub-page, the alternates.languages dictionary inherited from the home layout maps each locale to that locale's homepage, not the equivalent localized slug. So /fr/reserver-une-demo declares hreflang="de" targeting /de instead of /de/demo-buchen. Google will either ignore these hreflang declarations or treat the cross-locale relationships as broken.

Same fix as Critical 1: when adding generateMetadata per sub-route, also rebuild alternates.languages for each locale using getLocalizedPath(routeKey, locale), not getLocalizedPath("home", locale).

Critical 3   <html lang="en"> hardcoded across all locales

The root layout (app/layout.tsx) hardcodes <html lang="en">. The locale-specific <div lang={locale}> child does not compensate for this in the eyes of Google or accessibility tooling. On /fr, /he, /de, etc. Google reads lang="en" at SSR. The post-hydration useEffect that mutates document.documentElement.lang on the client is too late for SEO.

Fix: Move <html> from the root layout into app/[locale]/layout.tsx (the root must still render <html> per Next.js convention; the cleanest workaround in Next 14 is to keep the root as a pass-through and place real <html lang={locale}> inside the locale layout, or alternatively read the path via headers() in the root layout to compute the right lang at SSR).

Critical 4   robots.txt returns HTTP 404 (HTML, not text/plain)

The Next.js [locale] catch-all route is intercepting /robots.txt before any file-system fallback can serve a real robots.txt. Result: GET /robots.txt returns 404 with content-type: text/html and x-matched-path: /[locale]. There is no crawl directive, no sitemap declaration, and no AI bot allow signal (GPTBot, OAI-SearchBot, ClaudeBot, PerplexityBot, Google-Extended, CCBot).

Fix: add app/robots.ts using the Next.js Metadata API. Sample below.

// app/robots.ts
import { MetadataRoute } from 'next'
export default function robots(): MetadataRoute.Robots {
  return {
    rules: [
      { userAgent: '*', allow: '/' },
      { userAgent: ['GPTBot', 'OAI-SearchBot', 'ClaudeBot', 'PerplexityBot', 'Google-Extended'], allow: '/' },
    ],
    sitemap: 'https://truecalling.app/sitemap.xml',
  }
}

Critical 5   Staging deployment is openly indexable

truecalling-lyart.vercel.app is live, returns meta name="robots" content="index, follow" on every page, and serves a sitemap that references the production domain truecalling.app. Googlebot can crawl both — duplicate content risk and PageRank dilution.

Fix: in next.config.js, add a X-Robots-Tag: noindex header conditional on process.env.VERCEL_ENV !== 'production'. Reinforce in generateMetadata() with robots: { index: false } on the same condition.

SXO findings (41 / 100)

Critical finding   Page-type mismatch on primary keywords

9 of 10 top results for "AI sourcing software" are comparison listicles or buyer's guides. The TrueCalling homepage is a SaaS landing page. Google has formed an unambiguous editorial consensus: for these queries, it rewards third-party comparison content, not vendor homepages.

The homepage cannot rank for "AI sourcing software" or "recruiting AI tool" regardless of on-page optimization. The fix is not rewriting the home — it is creating a separate comparison guide asset (4-6 k words) that ranks for these queries; the homepage is reserved for branded and bottom-of-funnel traffic.
